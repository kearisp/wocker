# Розгортання проєкту на прикладі

У цьому прикладі розгорнемо проєкт за допомогою **wocker** на базі репозиторію
[timer](https://github.com/kearisp/timer). Пройдемо шлях від клонування до запуску стеку, пояснимо призначення кожної
команди та наведемо кілька порад із діагностики.

## 1) Клонування репозиторію

```shell
git clone https://github.com/kearisp/timer.git
cd timer
```

- Локально завантажуємо код і переходимо до директорії проєкту.

## 2) Ініціалізація wocker для проєкту

```shell
ws init
```

- Генерує базову конфігурацію `.wocker` (контейнери, сервіси, шаблони оточення).
- За можливості виявляє стек та пропонує типові налаштування.
- Команду можна запускати повторно — вона не видалить вашу роботу.

Детальніше: [ініціалізація проєкту](/docs/project/init).

## 3) Монтування коду в контейнер

```shell
ws volume:mount ./:/var/www/html
```

- Прив’язує поточну директорію (`./`) до `/var/www/html` у контейнері.
- Дозволяє live‑reload: зміни в коді одразу видно всередині сервісів.
- Переглянути всі монтування:

```shell
ws volumes
```

## 4) Запуск проєкта

```shell
ws start
```

- За потреби витягує/збирає образи.
- Створює мережі, томи й запускає всі сервіси.
- Перший запуск може тривати довше через завантаження залежностей.

### Перевірка стану та логів

Після `ws start` перевірте здоров’я сервісів:

```shell
ws ps                     # список запущених проєктів
ws logs                   # логи проєкту в поточній директорії
ws logs -n=<project-name> # логи проєкту за його ім'ям (можна запускати з будь-якої директорії)
```

Якщо щось не стартує, скористайтеся діагностичними командами:

- `ws logs` — перегляд логів проєкту в поточній директорії.
- `ws logs -n=<project-name>` — перегляд логів проєкту за іменем з будь-якої директорії.
- `ws ps` — перевірка, які проєкти запущені.

### Зупинка та прибирання

```shell
ws stop          # зупинка проєкту
```

> Порада: використовуйте `ws restart` після змін у конфігурації.
